# iOS End-to-End Encryption Design

## Overview

Add E2E encryption support to the Phosphor iOS viewer app, matching the existing web client implementation. When a CLI session is started with `--key`, the iOS app must prompt for the passphrase, derive the same AES-256-GCM key, and decrypt/encrypt all terminal data.

No changes to the relay, CLI, or web client are required.

## Cryptographic Primitives

All parameters must match the Go and web implementations exactly for interoperability:

- **Key derivation**: PBKDF2-SHA256, 100,000 iterations, 32-byte output
- **Encryption**: AES-256-GCM
- **Salt**: 16 bytes, generated by CLI, sent base64-encoded in protocol messages
- **Nonce**: 12 bytes, randomly generated per message
- **Ciphertext format**: `[12-byte nonce][ciphertext + 16-byte GCM auth tag]`

### Implementation

**New file: `Services/CryptoManager.swift`**

A stateless utility struct with three functions:

- `deriveKey(passphrase: String, salt: Data) -> SymmetricKey` — PBKDF2 via CommonCrypto (`CCKeyDerivationPBKDF`), returns a 256-bit `CryptoKit.SymmetricKey`.
- `encrypt(key: SymmetricKey, plaintext: Data) throws -> Data` — `AES.GCM.seal()`, returns `nonce + ciphertext + tag` as a contiguous `Data`.
- `decrypt(key: SymmetricKey, data: Data) throws -> Data` — Splits input at byte 12, constructs `AES.GCM.SealedBox(nonce:ciphertext:tag:)` from the combined portion, calls `open()`.

Dependencies: `CryptoKit` (AES-GCM) and `CommonCrypto` (PBKDF2). Both are Apple frameworks — no third-party dependencies.

## Protocol Updates

**File: `Protocol/ProtocolCodec.swift`**

Add two optional fields to `JoinedPayload`:

```swift
struct JoinedPayload: Decodable {
    let mode: String
    let cols: Int
    let rows: Int
    let command: String
    let encrypted: Bool?          // new
    let encryptionSalt: String?   // new, base64-encoded, maps to "encryption_salt"
}
```

Use `CodingKeys` to map `encryption_salt` JSON key to `encryptionSalt` Swift property.

No new message types are needed. The existing `Joined` message (type `0x13`) already carries these fields from the relay — the iOS codec just doesn't parse them yet.

## WebSocket Updates

**File: `Services/WebSocketManager.swift`**

Add a new event variant:

```swift
enum WebSocketEvent {
    // existing cases...
    case encryptionRequired(salt: String)
}
```

Changes to message handling:

- When a `Joined` message is received with `encrypted == true`: emit `.encryptionRequired(salt:)` with the base64 salt string. Do not transition to `.connected` yet.
- Add a mutable `encryptionKey: SymmetricKey?` property. When set, the manager applies encryption/decryption to data messages.
- **Stdout receive path**: if `encryptionKey` is set, call `CryptoManager.decrypt()` on the payload before forwarding via `.stdout(Data)`. On failure, emit a new `.decryptionFailed` event.
- **Stdin send path**: if `encryptionKey` is set, call `CryptoManager.encrypt()` on the data before encoding and sending.

Add a method `setEncryptionKey(_ key: SymmetricKey)` that stores the key and transitions connection state to `.connected`.

## ViewModel Updates

**File: `ViewModels/TerminalViewModel.swift`**

New published state:

```swift
enum EncryptionState {
    case none
    case passphraseRequired(salt: String)
    case unlocked
    case failed  // shown after a decryption failure
}

@Published var encryptionState: EncryptionState = .none
```

Private stored property: `var encryptionKey: SymmetricKey?` (in-memory only, never persisted).

Behavior:

- On `.encryptionRequired(salt:)` event from WebSocket: set `encryptionState = .passphraseRequired(salt:)`.
- `submitPassphrase(_ passphrase: String, salt: String)`: derive key via `CryptoManager.deriveKey()`, store in `encryptionKey`, pass to `WebSocketManager.setEncryptionKey()`, set `encryptionState = .unlocked`.
- On `.decryptionFailed` event: clear `encryptionKey`, set `encryptionState = .failed` (which re-shows the passphrase view with an error message).
- Unencrypted sessions: `encryptionState` stays `.none`, no passphrase view shown.

## Passphrase View

**New file: `Views/PassphraseView.swift`**

A SwiftUI view displayed when `encryptionState` is `.passphraseRequired` or `.failed`. Matches the Phosphor dark terminal aesthetic.

Layout:
- Dark background (matching `PhosphorTheme`)
- Lock icon (`systemName: "lock.fill"`) in green
- Title: "Encrypted Session"
- Subtitle: "Enter the passphrase to decrypt this session"
- `SecureField` with green border
- "Unlock" button
- Error label (visible only in `.failed` state): "Incorrect passphrase. Try again."

The view calls `TerminalViewModel.submitPassphrase()` on form submission.

### Integration in `TerminalContainerView`

The existing `TerminalContainerView` switches on `encryptionState`:
- `.none` or `.unlocked`: show the terminal
- `.passphraseRequired` or `.failed`: show `PassphraseView`

## Data Flow

```
1. User taps session in SessionListView
2. TerminalViewModel.connect() opens WebSocket, sends Join message
3. Relay responds with Joined(encrypted=true, encryption_salt="base64...")
4. WebSocketManager emits .encryptionRequired(salt:)
5. TerminalViewModel sets encryptionState = .passphraseRequired(salt:)
6. TerminalContainerView shows PassphraseView
7. User enters passphrase, taps Unlock
8. TerminalViewModel.submitPassphrase() derives key, stores in memory
9. WebSocketManager receives key, transitions to .connected
10. Stdout arrives → WebSocketManager decrypts → TerminalViewModel → terminal UI
11. User types → TerminalViewModel → WebSocketManager encrypts → relay → CLI
12. On decryption failure: clear key, show PassphraseView with error
```

## Key Caching

Derived key is held in-memory only (`SymmetricKey` property on `TerminalViewModel`). It is lost when:
- The view is dismissed
- The app is terminated
- A decryption failure occurs (explicit clear)

No Keychain or disk persistence. This matches the web client's `sessionStorage` approach.

## Error Handling

- **Wrong passphrase**: The first decrypted stdout message will fail AES-GCM authentication. On failure, the key is cleared and the passphrase view is re-shown with an error message.
- **Unencrypted session with encryption fields absent**: `encrypted` defaults to `nil`/`false`, passphrase view is never shown. No behavior change.
- **Encryption key set but message is not encrypted**: This cannot happen — encryption is session-wide. If the CLI sends encrypted, all messages are encrypted.
