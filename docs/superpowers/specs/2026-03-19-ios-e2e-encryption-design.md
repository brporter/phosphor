# iOS End-to-End Encryption Design

## Overview

Add E2E encryption support to the Phosphor iOS viewer app, matching the existing web client implementation. When a CLI session is started with `--key`, the iOS app must prompt for the passphrase, derive the same AES-256-GCM key, and decrypt/encrypt all terminal data.

No changes to the relay, CLI, or web client are required.

## Cryptographic Primitives

All parameters must match the Go and web implementations exactly for interoperability:

- **Key derivation**: PBKDF2-SHA256, 100,000 iterations, 32-byte output
- **Encryption**: AES-256-GCM
- **Salt**: 16 bytes, generated by CLI, sent base64-encoded in protocol messages
- **Nonce**: 12 bytes, randomly generated per message
- **Ciphertext format**: `[12-byte nonce][ciphertext + 16-byte GCM auth tag]`

### Implementation

**New file: `Services/CryptoManager.swift`**

A stateless utility struct with three functions:

- `deriveKey(passphrase: String, salt: Data) -> SymmetricKey` â€” PBKDF2 via CommonCrypto (`CCKeyDerivationPBKDF`), returns a 256-bit `CryptoKit.SymmetricKey`.
- `encrypt(key: SymmetricKey, plaintext: Data) throws -> Data` â€” `AES.GCM.seal()`, returns `nonce + ciphertext + tag` as a contiguous `Data`.
- `decrypt(key: SymmetricKey, data: Data) throws -> Data` â€” Guards that input is at least 28 bytes (12 nonce + 16 tag minimum), then passes the full blob to `AES.GCM.SealedBox(combined: data)` which expects `nonce + ciphertext + tag` in that order. Calls `AES.GCM.open()` to decrypt.

All functions are pure and stateless â€” safe to call from any isolation context. The struct should conform to `Sendable`.

Dependencies: `CryptoKit` (AES-GCM) and `CommonCrypto` (PBKDF2). Both are Apple frameworks â€” no third-party dependencies.

## Protocol Updates

**File: `Protocol/ProtocolCodec.swift`**

Add two optional fields to `JoinedPayload`:

```swift
struct JoinedPayload: Decodable {
    let mode: String
    let cols: Int
    let rows: Int
    let command: String
    let encrypted: Bool?          // new
    let encryptionSalt: String?   // new, base64-encoded, maps to "encryption_salt"
}
```

Use `CodingKeys` to map `encryption_salt` JSON key to `encryptionSalt` Swift property.

No new message types are needed. The existing `Joined` message (type `0x13`) already carries these fields from the relay â€” the iOS codec just doesn't parse them yet.

## WebSocket Updates

**File: `Services/WebSocketManager.swift`**

Add a new event variant:

```swift
enum WebSocketEvent {
    // existing cases...
    case encryptionRequired(salt: String)
}
```

Changes to message handling:

- When a `Joined` message is received with `encrypted == true`: emit `.encryptionRequired(salt:)` with the base64 salt string. Do not transition to `.connected` yet.
- Add mutable properties: `encryptionKey: SymmetricKey?` and `bufferedEncryptedStdout: [Data]` (array of raw encrypted chunks).
- **Stdout receive path (no key yet)**: if session is encrypted but `encryptionKey` is nil, append the raw payload to `bufferedEncryptedStdout`.
- **Stdout receive path (key set)**: call `CryptoManager.decrypt()` on the payload before forwarding via `.stdout(Data)`. On failure, emit `.decryptionFailed(rawData: Data)` carrying the failed chunk so the ViewModel can re-buffer it for retry.
- **Stdin send path**: if `encryptionKey` is set, call `CryptoManager.encrypt()` on the data before encoding and sending.

Add a method `setEncryptionKey(_ key: SymmetricKey)` that stores the key, then decrypts and flushes all `bufferedEncryptedStdout` chunks in order (emitting `.stdout` for each). After flushing, transitions connection state to `.connected`.

## ViewModel Updates

**File: `ViewModels/TerminalViewModel.swift`**

New published state:

```swift
enum EncryptionState {
    case none
    case passphraseRequired(salt: String)
    case unlocked
    case failed  // shown after a decryption failure
}

@Published var encryptionState: EncryptionState = .none
```

Private stored property: `var encryptionKey: SymmetricKey?` (in-memory only, never persisted).

Behavior:

- On `.encryptionRequired(salt:)` event from WebSocket: set `encryptionState = .passphraseRequired(salt:)`.
- `submitPassphrase(_ passphrase: String, salt: String)`: derive key via `CryptoManager.deriveKey()`. Before committing: if there are buffered stdout chunks, trial-decrypt the first one. If trial decryption fails, set `encryptionState = .failed` immediately (wrong passphrase) without storing the key. If it succeeds (or no buffered chunks yet), store in `encryptionKey`, pass to `WebSocketManager.setEncryptionKey()` (which flushes buffered data), set `encryptionState = .unlocked`.
- On `.decryptionFailed(rawData:)` event: clear `encryptionKey`, re-buffer the failed chunk in `WebSocketManager.bufferedEncryptedStdout`, set `encryptionState = .failed` (which re-shows the passphrase view with an error message).
- Unencrypted sessions: `encryptionState` stays `.none`, no passphrase view shown.

## Passphrase View

**New file: `Views/PassphraseView.swift`**

A SwiftUI view displayed when `encryptionState` is `.passphraseRequired` or `.failed`. Matches the Phosphor dark terminal aesthetic.

Layout:
- Dark background (matching `PhosphorTheme`)
- Lock icon (`systemName: "lock.fill"`) in green
- Title: "Encrypted Session"
- Subtitle: "Enter the passphrase to decrypt this session"
- `SecureField` with green border
- "Unlock" button
- Error label (visible only in `.failed` state): "Incorrect passphrase. Try again."

The view calls `TerminalViewModel.submitPassphrase()` on form submission.

### Integration in `TerminalContainerView`

The existing `TerminalContainerView` switches on `encryptionState`:
- `.none` or `.unlocked`: show the terminal
- `.passphraseRequired` or `.failed`: show `PassphraseView`

## Data Flow

```
1. User taps session in SessionListView
2. TerminalViewModel.connect() opens WebSocket, sends Join message
3. Relay responds with Joined(encrypted=true, encryption_salt="base64...")
4. WebSocketManager emits .encryptionRequired(salt:)
5. TerminalViewModel sets encryptionState = .passphraseRequired(salt:)
6. TerminalContainerView shows PassphraseView
7. User enters passphrase, taps Unlock
8. TerminalViewModel.submitPassphrase() derives key, trial-decrypts first buffered chunk
9. If trial fails: show error, stay on PassphraseView. If succeeds: store key.
10. WebSocketManager.setEncryptionKey() flushes buffered stdout (decrypt + emit each)
11. WebSocketManager transitions to .connected
12. Subsequent stdout arrives â†’ WebSocketManager decrypts â†’ TerminalViewModel â†’ terminal UI
13. User types â†’ TerminalViewModel â†’ WebSocketManager encrypts â†’ relay â†’ CLI
14. On decryption failure: clear key, re-buffer failed chunk, show PassphraseView with error
```

## Key Caching

Derived key is held in-memory only (`SymmetricKey` property on `TerminalViewModel`). It is lost when:
- The view is dismissed
- The app is terminated
- A decryption failure occurs (explicit clear)

No Keychain or disk persistence. This matches the web client's `sessionStorage` approach.

## Error Handling

- **Wrong passphrase**: Detected at two points: (1) in `submitPassphrase()` via trial decryption of the first buffered chunk â€” immediate rejection without entering unlocked state; (2) at runtime if a later message fails decryption â€” key is cleared, failed chunk is re-buffered, and the passphrase view is re-shown.
- **Unencrypted session with encryption fields absent**: `encrypted` defaults to `nil`/`false`, passphrase view is never shown. No behavior change.
- **Encryption key set but message is not encrypted**: This cannot happen â€” encryption is session-wide. If the CLI sends encrypted, all messages are encrypted.

## Additional Code Fixups

**`.mode` event handler in `TerminalViewModel.swift`**: The existing handler reconstructs `JoinedPayload` without the new fields. It must carry forward `encrypted` and `encryptionSalt` when rebuilding the payload.
